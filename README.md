# Task Processing Pipeline

Task Processing Pipeline is a Bun + Express + PostgreSQL + Redis service for receiving signed webhooks, storing them as jobs, processing them through a pipeline action, and delivering the processed result to downstream subscribers.

## Setup

### Prerequisites

- Bun
- Docker and Docker Compose
- PostgreSQL and Redis via Docker Compose or equivalent local services

### Environment

Copy `example.env` into `.env` and adjust values if needed.

### Start infrastructure

```bash
docker compose up -d
```

This starts:

- Express API on `localhost:8001`
- PostgreSQL on `localhost:5432`
- Redis on `localhost:6379`

The API service is configured to depend on healthy PostgreSQL and Redis containers before it starts.

### Run the application

The API runs inside Docker Compose by default.

To run it locally outside Docker:

Development mode:

```bash
bun run dev
```

Normal start:

```bash
bun run start
```

The server runs database migrations on startup before accepting requests.

### Useful scripts

```bash
bun run lint
bun run db:generate
bun run db:migrate
bun run workers
bun run workers:processor
bun run workers:delivery
```

## DB Schema

The schema is defined with Drizzle in `src/db/schema`.

### `pipelines`

Represents a logical processing pipeline and its action.

Fields:

- `id`: UUID primary key
- `name`: human-readable pipeline name
- `source_path`: unique webhook slug used to route incoming webhook traffic
- `secret`: shared secret used for webhook signature verification
- `action_type`: one of `transform`, `filter`, `enrich`
- `action_config`: JSON configuration validated against the selected `action_type`
- `description`: optional description
- `created_at`, `updated_at`: timestamps

Supported action configs:

### `transform`

```json
{
  "field_mappings": {
    "firstName": "first_name",
    "lastName": "last_name"
  },
  "remove_fields": ["debug", "internal"]
}
```

- `field_mappings`: map existing field names to new field names
- `remove_fields`: remove fields from the payload after mapping

### `filter`

```json
{
  "mode": "all",
  "conditions": [
    {
      "field": "amount",
      "operator": "gte",
      "value": 100
    },
    {
      "field": "status",
      "operator": "eq",
      "value": "approved"
    }
  ]
}
```

- `mode`: `all` or `any`
- `conditions`: one or more condition objects
- `operator`: `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `contains`, `in`

### `enrich`

```json
{
  "fields": {
    "source": "crm",
    "version": 1
  },
  "overwrite_existing": true
}
```

- `fields`: static values added to the payload
- `overwrite_existing`: controls whether existing keys can be overwritten

### `jobs`

Represents one unit of pipeline work generated by an incoming webhook.

Fields:

- `id`: UUID primary key
- `pipeline_id`: foreign key to `pipelines.id`
- `status`: one of `pending`, `processing`, `processing-failed`, `processed`, `delivery-failed`, `completed`
- `payload`: original webhook payload
- `result`: processing result produced by the processor worker
- `retry_count`: processor retry count
- `subscriber_count`: number of subscribers targeted during delivery
- `total_deliveries`: number of delivery attempts made
- `completed_at`: completion timestamp
- `created_at`, `updated_at`: timestamps

Indexes:

- partial index for pending jobs
- partial index for processed jobs
- partial index for processing-failed jobs

### `subscribers`

Represents downstream webhook consumers for a given pipeline.

Fields:

- `id`: UUID primary key
- `pipeline_id`: foreign key to `pipelines.id`
- `url`: subscriber endpoint
- `secret`: shared secret used to sign delivery requests
- `created_at`, `updated_at`: timestamps

### `delivery_attempts`

Tracks every attempt to deliver a processed job to a subscriber.

Fields:

- `id`: UUID primary key
- `job_id`: foreign key to `jobs.id`
- `subscriber_id`: foreign key to `subscribers.id`
- `attempt_number`: current attempt number
- `response_status`: HTTP status from subscriber
- `response_body`: response payload captured from subscriber
- `error`: transport or application error text
- `status`: `pending`, `delivered`, or `failed`
- `scheduled_for`: timestamp when the attempt was scheduled
- `delivered_at`: timestamp for successful delivery
- `created_at`, `updated_at`: timestamps

Returns all pipelines.

Creates a pipeline. `action_config` is validated against `action_type`.

```json
  "source_path": "customer-created",

    "remove_fields": ["debug"]
  },

}
```

Example `filter` request:

```json
{
  "name": "Approved Orders",
  "source_path": "orders-approved",
  "secret": "super-secret",
  "action_type": "filter",

    "mode": "all",
    "conditions": [

      },
```

Deletes a subscriber.

### Incoming Webhooks

#### `POST /api/v1/webhooks/pipelines/:slug`

Receives a signed webhook for a pipeline identified by `source_path`.

Headers:

- `content-type: application/json`
- `x-pipeline-signature: <signature>`

Behavior:

- request body is parsed as raw JSON with a `256kb` limit
- signature is verified using the pipeline secret
- a new job is inserted with status `pending`
- response is `202 Accepted`

Example request body:

```json
{
  "customerId": "cus_123",
  "status": "approved",
  "amount": 150
}
```

## Architecture

The system is split into API, persistence, and worker responsibilities.

### Request flow

1. A client creates a pipeline and registers subscribers.
2. An external system sends a signed webhook to `/api/v1/webhooks/pipelines/:slug`.
3. The API verifies the signature, stores the payload as a `pending` job, and enqueues a processor task in BullMQ.
4. The processor worker consumes BullMQ tasks, atomically claims pending jobs in Postgres, and applies the pipeline action.
5. The processor stores the action output in `jobs.result`, marks the job as `processed`, then enqueues a delivery seed task in BullMQ.
6. The delivery worker fans out one BullMQ task per subscriber, retries only the failed subscriber task, logs every attempt, and marks the parent job as `completed` or `delivery-failed`.

### Worker model

- `processor` worker handles business transformation of the webhook payload
- `delivery` worker handles outbound fan-out and per-subscriber retry logic
- both workers are event-driven via BullMQ (no interval polling loop)
- BullMQ uses the existing Redis deployment as the queue backend
- Postgres remains the source of truth for job state and audit records
- processor-side job claiming still uses SQL locking semantics to avoid duplicate processing across worker instances

### Processing model

Pipeline processing is action-based:

- `transform`: rename or remove fields
- `filter`: evaluate the payload against configured conditions
- `enrich`: add static fields into the payload

The processor uses separate action handlers instead of a single large switch-heavy function. This keeps behavior isolated per action type and makes adding a new pipeline action straightforward.

## Design Decisions

### Bun for runtime and tooling

The project uses Bun for application startup and local development scripts. This keeps the runtime simple and reduces external tooling overhead.

### PostgreSQL as the source of truth

Jobs, pipelines, subscribers, and delivery attempts are persisted in PostgreSQL. This provides durable workflow state and makes retries and auditing straightforward.

### BullMQ on top of existing Redis

The system now uses BullMQ for processor and delivery orchestration, backed by the Redis instance already required by the service. This avoided introducing a second broker and let us reuse existing operational tooling and infrastructure.

### Event-driven workers instead of polling

Workers no longer wake up on fixed intervals to scan for work. They consume queue events immediately, which reduces idle DB load and shortens webhook-to-processing latency.

### Hybrid queue + database state model

BullMQ controls dispatch, delay, and retry timing, while Postgres remains the durable source of truth for workflow status and delivery history. This combines low-latency worker wakeups with strong auditability.

### Subscriber-level delivery retries

Delivery is modeled as one queue task per subscriber. A failing subscriber retries independently, so successful subscribers are not re-delivered on every retry cycle.

### Validation at the API boundary

Pipeline creation and updates validate `action_config` against `action_type` before data reaches the database. This prevents invalid pipeline definitions from entering the system and failing later inside workers.

### Separate processing and delivery phases

Processing and outbound delivery are intentionally split into different workers. That separation keeps transformation concerns independent from network I/O, retry logic, and subscriber-specific failures.

### Signed webhooks on ingress and egress

Incoming webhooks are verified with `x-pipeline-signature`, and outbound deliveries are signed with `x-delivery-sign`. This gives both sides a shared-secret verification model.

### Limited raw webhook payload size

Incoming webhook bodies are limited to `256kb` for `application/json`. This reduces abuse risk and prevents oversized payloads from entering the pipeline.
